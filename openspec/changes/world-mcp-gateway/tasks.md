## 2. mcp-world-server

- [ ] 2.1 Implement an `rmcp` server that runs alongside the sim and advertises its full tool + resource surface over the standard MCP protocol; test that a conforming client discovers the tool and resource list with no custom client code or custom wire format (mcp-world-server "Standard MCP server runs alongside the sim" → "A standard MCP client discovers the surface")
- [ ] 2.2 Bind the server's lifetime to the sim host: it starts when the sim is hosted and stops accepting requests once the sim stops; test the reachable-while-running / unreachable-after-stop lifecycle (mcp-world-server scenario "Server lifetime is bound to the sim"; Migration Plan Rollout/Rollback)
- [ ] 2.3 Serve every read from the latest `WorldSnapshot` and treat every write as an intent enqueued into the existing tick pipeline; test that a resource read mutates no sim state and a tool invocation enqueues an intent and returns immediately (mcp-world-server "Server owns the snapshot/intent contract" → both scenarios; design Decision 3)
- [ ] 2.4 Guarantee the server never stalls, waits on, or drops ticks: handling any request is independent of tick cadence; test that tick cadence is unaffected under high/slow/many concurrent request volume (mcp-world-server "Server never blocks the tick loop" → "Tick cadence is unaffected by request volume"; design Decision 3)
- [ ] 2.5 Ensure a blocked/stuck client cannot freeze the world or other clients: test that a client that stops reading mid-response leaves the tick loop advancing and other clients unblocked (mcp-world-server scenario "A blocked client does not freeze the world")
- [ ] 2.6 Enforce explicit, bounded `t+1` action latency at the server: an intent observed at tick `t` applies no earlier than `t+1`, applied only by the sim on its own schedule and never fast-tracked/short-circuited into the current tick by the request handler; test both the `t+1`-visibility and "the sim, not the server, applies the intent" scenarios (mcp-world-server "Action latency is explicit and bounded"; design Decision 3)

## 3. mcp-observation-resources

- [ ] 3.1 Expose `world://self` projecting only the requesting agent's avatar self-state (position-derived readout, facing/orientation, health/condition, weapon/loadout) from the latest `WorldSnapshot`; test it carries no other agent's private state and reflects the most recent snapshot, not a stale one (mcp-observation-resources "Self-state observation resource" → both scenarios)
- [ ] 3.2 Expose `world://nearby` returning AOI entities each with entity type, bearing relative to the requesting avatar, and a NUMERIC relative distance, derived from the latest `WorldSnapshot`; test the type/bearing/numeric-distance shape and that only legitimately perceptible entities appear (mcp-observation-resources "Nearby-entities observation resource" → "Nearby list carries type, bearing, and numeric distance")
- [ ] 3.3 Test that an entity present in the snapshot but outside the requesting avatar's LOS/AOI is excluded from `world://nearby` (mcp-observation-resources scenario "Entities outside the area of interest are excluded")
- [ ] 3.4 Expose `world://summary` as an optional NL summary rendered over the SAME LOS/AOI-scoped snapshot data backing `self`/`nearby`; test it describes only perceptible state, mentions/implies nothing outside AOI, and is consistent with the structured resources (introduces no entity absent from the scoped `nearby` set) (mcp-observation-resources "Optional natural-language summary resource" → both scenarios)
- [ ] 3.5 Expose `world://events` as a low-frequency event-subscription doorbell for significant perceptible events (took-damage, perceptible geometry change, target spotted) that signals "come read the resources" and carries no per-frame world state; test the doorbell prompts a read rather than embedding state (mcp-observation-resources "Low-frequency event doorbell resource" → "Doorbell signals the agent to come read"; design Decision 4)
- [ ] 3.6 Test `world://events` is not a per-frame push channel — many ticks without a significant perceptible event emit no per-tick/per-frame notification (mcp-observation-resources scenario "Events resource is not a per-frame push channel"; design Decision 4)
- [ ] 3.7 Test doorbells respect AOI — a significant event outside the requesting avatar's LOS/AOI delivers no doorbell to that agent (mcp-observation-resources scenario "Doorbells respect the area of interest"; design Decision 5)
- [ ] 3.8 Enforce the observation payload as a server-side trust boundary: every resource (`self`/`nearby`/`summary`/`events`) is LOS/AOI-scoped on the way out of the snapshot, with no out-of-AOI state derivable from any payload; test that no resource leaks out-of-AOI state and that scoping ignores any client-supplied attempt to widen scope (mcp-observation-resources "Observation payload is a trust boundary…" → both scenarios; design Decision 5; CONTRACT §8)
- [ ] 3.9 Compute the LOS/AOI scope once and make obs-scope == action-auth scope: the entity/region set observable through the resources is bounded by the same scope used to authorize actions, and an entity outside every observation resource is outside the action-auth scope; test both "observation and action share one scope" and "unobservable targets are unactionable" (mcp-observation-resources "Observation scope equals action-authorization scope" → both scenarios; design Decision 5; CONTRACT §8)
- [ ] 3.10 Serve every observation read from the most recent completed `WorldSnapshot` immediately, never blocking on a live tick boundary; test that a read mid-/between-tick returns the latest snapshot without waiting or stalling the loop, and that concurrent reads never block each other or the sim (mcp-observation-resources "Reads come from the most recent snapshot without blocking the tick" → both scenarios; design Decision 3)

## 4. mcp-action-tools

- [ ] 4.1 Advertise exactly the tool set `move`, `turn`, `use`, `say` via `#[tool]` macros with `schemars`-auto-generated input schemas (the advertised schema == the schema the sim validates against); test the listed surface is exactly these four with no additional action tool (mcp-action-tools "Curated High-Level Action Tool Surface" → "Tool surface advertises exactly move/turn/use/say"; design Decisions 1, 6)
- [ ] 4.2 Bind `move`/`turn` to a movement/orientation intent and `use` to the action-key intent; test that a valid `move` invocation enqueues a movement intent into the tick pipeline (mcp-action-tools scenario "move submits a movement intent"; design Decision 2)
- [ ] 4.3 Bind `say` to a communication intent; test that a valid `say` invocation enqueues a communication intent (mcp-action-tools scenario "say submits a communication intent"; design Decision 2)
- [ ] 4.4 Exclude any `fire`/weapon/combat tool at the surface level by constitution — there is nothing to wire; test no `fire` and no weapon/combat tool is advertised, and an attempt to invoke `fire` is rejected as unknown/unsupported with no weapon/combat intent enqueued (mcp-action-tools "No Weapon or Fire Tool" → both scenarios; design Decision 6; CONTRACT §8)
- [ ] 4.5 Make each tool a non-blocking intent enqueue that returns after enqueueing without waiting for the intent to be applied; test the tool returns without waiting for a tick and the tick loop keeps advancing independently of the call (mcp-action-tools "Non-Blocking Intent Enqueue" → both scenarios; design Decision 3)
- [ ] 4.6 Validate every action server-side before enqueue and authorize it only within the requesting agent's obs/AOI scope (obs-scope == action-auth scope); test that a malformed argument is rejected with no intent enqueued, an action targeting a region outside the agent's AOI is rejected as out-of-scope with no intent enqueued, and a valid in-scope action passes validation and is enqueued (mcp-action-tools "Server-Side Validation Within Agent AOI Scope" → all three scenarios; design Decision 5; CONTRACT §8)
- [ ] 4.7 Make action effect timing explicit: an intent submitted during tick `t` applies on the sim's own schedule no earlier than `t+1` and has no effect on tick `t`'s world state; test the `t+1` timing scenario (mcp-action-tools "Explicit Action Effect Timing"; design Decision 3)
